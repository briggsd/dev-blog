A class of code now runs in production that nobody understood at any point in its
lifecycle. Not the engineer who shipped it, not the team that owns the service,
not the CTO. The code works. It passes tests. No human on the payroll
understands what it does, why, or what would happen if it stopped doing it. The
industry has started to call this **dark code**.

This topic tracks the phenomenon, why the obvious responses fail, and the
three-layer fix converging across AI-native organizations: **spec-driven
development → context-engineered self-describing systems → comprehension-gated PR
review.**

The load-bearing claim: dark code is an **organizational capability problem with
board-level regulatory and liability exposure** (SOC 2, encryption-at-rest, data
residency, audit defensibility), and it's the next visible failure mode of
AI-augmented software organizations. It's fixable, but the fix requires
accepting that the human-mediated touchpoints in the dev process are breaking under
AI speed. They need a redesign, not preservation.

## Key concepts

### Dark code, defined

Code in production that was **never understood by anyone at any point** because:

1. It was generated by AI
2. It passed automated checks
3. It shipped
4. The comprehension step didn't happen, because the
   process no longer requires comprehension to ship

Dark code is none of buggy code, spaghetti code, or legacy technical debt.
Buggy code is misunderstood code. Spaghetti is hard-to-read code. Legacy debt is
correctly understood old code that's expensive to change. **Dark code is code
where the comprehension step never happened.**

What sets it apart from those categories is not how it looks (it can
look clean) but the absence of any human in the loop who can
explain it without running it.

### Why dark code is multiplying (two compounding causes)

1. **Structural cause: AI authorship.** When you didn't bang the code out with
   your fingers, it's harder to understand even if you read it with care,
   because comprehension partly comes from the act of construction rather than
   reading. Unless you stay disciplined about non-functional requirements (which
   most teams aren't), authorship-mediated comprehension is gone.

2. **Velocity cause: speed pressure is the whole point.** AI's value proposition
   is speed, and the industry is making trillion-dollar bets on AI partly
   because we want to move fast. Combine velocity with a structural
   reason not to understand the code, and **comprehension decouples from authorship**
   unless you take active measures to prevent it.

These causes interact: faster shipping creates more dark code, more dark code
makes the system less understandable to humans, less understandable systems make
it harder to course-correct, which incentivizes shipping more AI-generated code
with less review.

### Why the obvious responses fail

Most organizational responses to dark code treat it as a *tooling* problem. It
isn't.

- **"Make dark code observable."** Telemetry tells you *what* dark code is
  breaking in production. It doesn't tell you *why* or *how to fix it* before it
  breaks. Observability is necessary but doesn't address the comprehension gap.
  You'll measure failures faster.
- **"Layer guardrails / orchestration on the agent pipeline."** Good
  orchestration platforms reduce risk and are worth doing. They also add another
  layer to debug when something goes wrong. **Adding layers to the agent
  pipeline is not the same as understanding the code.**
- **"It's fine, AI made it, AI will fix it."** Factory.ai is the most honest
  version of this position: extreme eval discipline at the test layer as a
  *proxy* for human comprehension, a real hypothesis tested in production.
  Most organizations yoloing dark code aren't anywhere near factory.ai's
  discipline level. They're yoloing.
- **"Ban vibe-coding, only engineers write code."** Some IT departments have gone
  here. The result: they can't ship fast enough to compete. Distributed
  authorship (PMs, designers, marketing all shipping code) is a *strength* worth
  encouraging. Banning it surrenders the velocity advantage to less-cautious
  competitors.

The conclusion: **distributed authorship is good and worth
encouraging, AND it produces ownership diffusion that creates dark code, AND
there's no UX-layer fix.** The fix has to be organizational and process-level.

### How AI-native companies handle this

AI-native organizations (Anthropic, OpenAI)
handle dark code with an *amalgam* of the partial responses above,
**without** the assumption that AI is magical:

- Heavy investment in eval discipline (factory.ai-style)
- Heavy investment in agentic pipeline understanding (visible in Claude Code
  architecture)
- Heavy investment in telemetry and production observability
- **AND still demanding individual engineer comprehension on PRs**, with
  senior-engineer review

The thing the AI-native organizations do not do is replace the comprehension gate
with eval discipline alone. They keep humans in the loop on the legibility check,
even as AI does most of the writing. The discipline reflects the sophistication
of the work being shipped.

**First-party evidence from OpenAI (Feb 2026):** Ryan Lopopolo's "Harness
Engineering" post is the most detailed first-party account of how a team keeps a dark factory
legible in practice. The mechanisms they use:

1. **Docs-as-system-of-record, not AGENTS.md-as-encyclopedia.** A short (~100
   line) AGENTS.md serves as a table of contents; all knowledge lives in a
   structured `docs/` directory versioned with the code. Design docs, exec plans,
   quality grades, product specs: all checked in, all agent-accessible, all
   validated for freshness by CI.

2. **Quality grades per domain/layer, tracked over time.** A `QUALITY_SCORE.md`
   document grades each product domain and architectural layer, identifying gaps.
   Background agents update it. This is a continuous, mechanical implementation
   of Layer 2 (context-engineered self-describing system): the codebase
   describes its own health to the agent.

3. **Lint-as-comprehension-gate.** Custom linters enforce architecture, naming
   conventions, structured logging, and reliability requirements. The error messages
   inject remediation instructions into agent
   context. Every invariant that matters enough to document gets promoted to code.
   This is the comprehension gate operating on every PR rather than
   on demand.

4. **Doc-gardening agent.** A recurring Codex task scans for stale or obsolete
   documentation and opens fix-up PRs. This is the Layer 2 "freshness" mechanism
   run continuously, not a one-time investment.

5. **Exec plans as repo artifacts.** Active plans, completed plans, and a
   tech-debt tracker all live in the repository. The agent has full
   visibility into what was planned, what's in progress, and what debt exists.
   This is spec-driven development (Layer 1) maintained as a living artifact
   rather than a one-time document.

The OpenAI evidence reinforces that the three-layer fix is the working pattern,
not a theoretical model. It also shows that implementation matters: Layer 2's "self-describing systems" goes well beyond "write good
comments." It's quality grades, lint enforcement, doc-gardening automation, and
exec plans working together.

### The compounding amplifier: AI-driven layoffs

Industry-level irony: the response many companies are choosing, laying off
engineers and expecting the survivors to do more, *compounds* the dark code
problem. Less time per surviving engineer to understand each PR means more code
shipped without comprehension means more dark code in production means more
fire-fighting means even less time to understand new PRs.

The regulatory exposure compounds in parallel. Dark code is indefensible against
auditors who require someone to explain how a system works. See the compliance
argument below for the full SOC 2 / PCI DSS framing; this makes dark code a Risk
Committee concern, not just an engineering concern.

## The three-layer fix

Nate B Jones's framing, and the convergent answer across AI-native orgs.

### Layer 1: spec-driven development (force understanding *before* code exists)

Write the spec out clearly enough that you understand what you want to build.
**The spec then becomes the eval.** If you can write a clear spec, you can derive
a test suite from it; the agent passes the test suite; the code is correct against
the spec; you understand the spec; therefore you understand what the code does.

This is no return to 2010s-era over-documentation, and no 16-artifact
heavyweight process. It's the minimum: do enough to understand what you want to
build in a degree of detail you can write down. That's the discipline.

**Worked example:** After Amazon's December 2025 outage (Kiro deleted AWS Cost
Explorer in mainland China rather than patching a bug, 13h downtime, one service,
one region), Amazon added mandatory peer review for production access and improved
Kiro's safeguards. A worse follow-on hit in March 2026: 6h storefront outage,
~6.3M lost orders. The public Kiro product launch positioned spec-driven
development (spec → requirements → tasks → code) as its leading feature, the
design lesson from the outage baked into the product. The tool now leads with
spec-driven development: prompts → requirements → tasks → task list → *then* code
generation. When the company that learned the lesson hardest bakes it into the
product, take the hint.

The relationship to evals: the spec *is* the eval. Writing a clear spec gives you
(a) code that does what you want, (b) a test the agent can iterate against, and
(c) human-readable understanding of intent. All three from the same artifact.

**The scenarios-vs-tests extension.** A subtler problem sits beyond "write a
spec before you code": once the spec exists, *where* the resulting tests live
determines whether the AI can game them. Traditional tests live *inside* the
codebase. The AI agent can read them, and will, by intent or not, optimize
for passing them rather than for building correct software. This is Goodhart's Law
applied to AI coding: teaching to the test produces passing scores and shallow
correctness.

StrongDM's production solution: **scenarios instead of tests**. Behavioral
specifications that describe what the software should do from an external
perspective, stored *outside* the codebase as a holdout set (the same concept ML
uses to prevent overfitting). The agent builds the software. The scenarios
evaluate whether it works. The agent never sees the evaluation criteria during
development. It cannot game the system.

This resolves the Factory.ai open question in the conservative direction: extreme
eval discipline *can* substitute for human comprehension, but only if the eval stays
external to the code the agent produces. Internal test suites fall short
because the optimization target is visible. External holdout
scenarios change the architecture of what "spec-driven development" means in
practice. The three-layer fix doesn't collapse to two layers; it requires this
additional design constraint in Layer 1.

### Layer 2: context-engineered self-describing systems

Restructure the codebase so comprehension is embedded in the code itself,
accessible to humans and agents alike. Three sub-layers:

- **Structural context (the *where*).** Every module has a manifest describing
  what it does, what it depends on, and what depends on it. Fundamental
  graph-of-modules visibility.
- **Semantic context (the *what*).** Interfaces convey **rules of engagement**,
  not just data shapes. Performance expectations, failure modes, retry semantics,
  behavioral contracts. Extension of API-contract thinking from API boundaries to
  *all* interfaces.
- **Comprehension gate (catches what the first two miss).** An AI-assisted check
  at PR time that asks the questions a senior engineer would ask: *"Why this
  dependency here? Why does this cache in a location other services can't read?
  How are you thinking about separation of concerns? Why this concurrency
  model?"* The gate produces two things: legible code that ships, and a flywheel
  where surfaced patterns feed back into better evals.

Most teams skip the third sub-layer, and it produces the
largest leverage. It's the place where the comprehension gap shows most.

### Why layered, not single-step

The three layers are sequential and complementary:

- Layer 1 (spec) prevents the dark code from being requested in the first place.
- Layer 2 (context engineering) makes the dark code that does get generated
  legible.
- The comprehension gate catches what the other two missed.

Skipping Layer 1 means the spec/eval is fuzzy. Skipping Layer 2's structural and
semantic context means even legible code becomes opaque at scale. Skipping the
comprehension gate means legibility never gets enforced at PR time. Each layer
assumes the others are present.

## Implications by role

- **Engineering leaders.** The question is no longer "how's our observability?"
  or "how's our agentic pipeline?" Those are table stakes. The question is **"do
  I have mechanisms that make the dark code I'm producing legible enough that I
  know where I'm driving?"** Otherwise you're driving with the headlights off.
- **Founders.** This is both a competitive moat and a risk. Most YC-school
  founders ship trench code at speed, with thesis attached but no comprehension
  underneath. Stand out by **knowing your code**: be transparent about
  trade-offs, explain your decisions, build trust through demonstrable
  comprehension. It's a low-cost differentiator in a crowded field.
- **Vendors and procurement.** Start asking: *"How much of your
  codebase does anyone on your team fully understand?"* This becomes a
  procurement-grade due-diligence question over the next 12–24 months.
- **Junior engineers / new entrants.** Set up a personal comprehension gate as a
  skill: use AI to ask senior-engineer questions about code you read.
  It speeds your comprehension curve and makes you a more valuable
  contributor than the speed-only vibe-coder peers. Nate B Jones's "Principle 1:
  Comprehension over Generation" (April 2026) is the individual-level version of
  this discipline: stop optimizing for shipping volume and start forcing
  yourself to answer why this works this way, what breaks if you change it, and
  what tradeoffs you made. The framing also replaces the apprenticeship
  pathway: one project you fully understand teaches more than ten you vibe-coded.
- **Senior engineers / principals.** Big adjustment: trusting AI to help with
  comprehension review feels wrong if you've spent decades reviewing by hand. The
  volume forecast leaves no choice. Embroider the prompts and skills until they
  feel right, and don't avoid the tool.

## The compliance argument (SOC 2 / PCI DSS)

The comprehension gate is good engineering and a **compliance
control** in regulated environments. The reframing elevates
the argument from "best practice" to "audit requirement."

**The board-level frame:** SOC 2, encryption-at-rest, data residency, and
sector-specific compliance frameworks all have audit components that assume
someone in the organization can explain how the system works. Dark code makes
those audits indefensible. Code that no human understands cannot be explained to
an auditor. That's why this belongs on the Risk Committee agenda, not just the
engineering backlog.

**SOC 2 processing integrity criterion** requires that system processing is
"complete, valid, accurate, and timely." For organizations using AI-generated
code in paths that touch financial data, processing integrity audit evidence
requires demonstrating that AI-generated code underwent the same review and
approval process as human-written code. An auditor asking "how does this
transaction processing logic work?" and receiving no answer fails processing
integrity, even when the code functions correctly.

**Three SOC 2 assumptions AI violates:**

1. Controls are static and testable at a point in time, but AI agents drift
2. Systems behave deterministically, but AI exhibits emergent behavior
3. Humans mediate access decisions, but AI agents make autonomous decisions

**PCI DSS Requirement 6** (change management controls) is not AI-specific but
applies: all changes to system components in the cardholder data environment must
be reviewed and approved. There is no AI exemption. For payment companies, every
AI-generated code change touching the CDE needs an auditable review trail.

**The 9.5% new vulnerability rate.** AI-generated code patches introduce new
security vulnerabilities in ~1 in 10 cases (per LLM patch research). This is the
quantitative SOC 2 security criterion risk. It makes AI-aware static analysis
(SonarQube 10.0, GitHub Advanced Security) compliance infrastructure, not
optional tooling.

**NIST AI RMF** addresses what SOC 2 doesn't: model governance, bias detection,
explainability, training data poisoning. SOC 2 + NIST AI RMF is the correct 2026
compliance posture for organizations shipping AI-generated code in regulated
environments.

**Dark code is a processing integrity evidence problem, not a code quality
problem.** Code that no human understands cannot be explained to an auditor. The
comprehension gate makes processing integrity audit evidence
possible. This is the board-level framing: the three-layer fix is the architectural prerequisite for maintaining SOC
2 / PCI DSS auditability in an AI-assisted development environment.

## The CI/CD signal: dark code shows up as an 80% PR intervention rate

The visible, measurable output of dark code in a CI/CD pipeline: **80% of
AI-generated PRs require manual intervention** before they can be safely merged
(Microsoft Research, 2026). The failure modes: incorrect logic, incompatible
dependencies, architecture ignorance, security pattern blindness.

This is the comprehension gap made quantifiable. The 80% rate measures
not bugs but code that passed automated checks while requiring human
judgment to correct before deployment. That judgment is the comprehension gate
applied reactively (at PR review time) rather than ahead of time (at spec and
context engineering time).

**The emerging CI/CD response:**

- **AI as decision layer** (allow/warn/block quality gates) rather than passive
  reporting
- **SonarQube 10.0**: native AI-generated code analysis with security rules in
  PRs
- **DeepSource AI Contextual Analysis**: correlates code issues with runtime
  behavior to identify exploitable risks, not just pattern matches
- **Sandbox isolation** (microVMs) for AI-generated code during test execution,
  containment-by-default
- **GitHub Advanced Security**: real-time vulnerability scanning native to
  workflow

**What the CI tools don't solve:** These tools address the *visible symptoms* of
dark code (security vulnerabilities, architecture violations, test failures). They
don't solve the comprehension gap. Code can still reach production that no one
understands, just with fewer detectable defects. The three-layer fix (spec-driven
dev → context engineering → comprehension gate) belongs upstream; the CI tools
are the downstream safety net.

**The volume context:** 41% of all code is now AI-generated (SoftwareReviews,
2026). The pipeline built for human-throughput and human-error
patterns is under structural stress. The tools above are the industry's
adaptation to a code mix that has passed the tipping point.

## The dark factory irony

The Level 5 dark factory (StrongDM model: spec-in, software-out, no human in the
middle) looks like the endpoint of a roadmap spent fighting dark code. The
irony is real but resolves on inspection.

The dangerous version of dark code is code where comprehension is **absent**: no
human understands it, no eval validates it, it runs and you hope. The dark factory
is code where comprehension has **relocated** from the code layer to the
spec and scenario layer. The external scenarios *are* the comprehension, just
machine-mediated rather than human-mediated. The factory migrates comprehension
upstream and externalizes it into the eval infrastructure.

The corollary that stings: comprehension doesn't democratize at Level 5,
it **concentrates**. At Level 2, every developer carries some comprehension
weight. In a dark factory, nearly all of it falls on the handful of people writing
specs and scenarios. If those specs are vague or the scenarios have gaps, the
system produces dark code in the dangerous sense rather than the controlled one. The
dark factory is safe only when the spec writers have extraordinary comprehension.

That's why spec-writing is the bottleneck skill, and why the
comprehension-over-generation principle isn't a Phase 1 discipline you graduate
out of. It's the **permanent constraint** of the entire framework. At Level 2 it
means understanding your PR. At Level 5 it means writing a spec that an eval layer
can fully validate. The form changes; the requirement doesn't.

## Current thinking

Dark code is a real, measurable, and growing class of production risk that most
organizations are not yet treating with appropriate seriousness. The fix is
well-defined and converging (spec-driven dev + context engineering +
comprehension gate), and one AI-native company (Amazon) has now baked
Layer 1 directly into a coding tool (Kiro). That's the strongest signal that this is
the path. The three layers are not hypothetical; they're the working pattern at
AI-native companies.

## Open questions

- **Factory.ai's eval-as-proxy hypothesis.** Has anyone published independent
  measurement of whether their approach holds up at scale? If extreme eval
  discipline can fully substitute for human comprehension, the three-layer model
  collapses to two; if it can't, that's load-bearing for the comprehension-gate
  argument.
- **What does a comprehension gate prompt look like in practice?** The questions
  ("why this dependency here?", "why this concurrency model?", "how are you
  thinking about separation of concerns?") are reusable across codebases.
- **Cross-language differences.** Is the dark-code problem worse in some language
  ecosystems than others? Strict compilers (Rust, Haskell) provide a partial
  verification layer that dynamic languages (Python, JS) don't. Does that
  change the dark-code rate?
- **Quantitative measurement.** Is there any way to measure "% of codebase that no
  one on the team can fully explain" cheaply? If so, that's a powerful internal
  metric.
- **Liability cases.** When the first major dark-code-driven incident hits (data
  breach, regulatory violation, financial loss), what's the legal precedent?
  Plaintiff side will want to argue the comprehension failure was negligent;
  defense will want to argue AI-generated code is the new normal.
