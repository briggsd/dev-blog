The harness is everything wrapped around the model: the orchestration loop, the tool layer, memory and state, context construction, output parsing, error handling, the verification loops, and subagent routing. The model is the reasoning substrate; the harness is the software that turns a raw LLM into a goal-directed, tool-using, self-correcting agent. The working definition, the model-versus-harness split, and Böckeler's guides-versus-sensors taxonomy are developed in [Agent Design](/working-intel/topics/agent-design/); this topic specializes in the envelope itself: how to grow one, how to instrument it, how to verify it, and where the field is going. The production substrate it runs on (sessions, sandboxes, the shared tool layer, harness sovereignty) lives in [Agent Infrastructure](/working-intel/topics/agent-infrastructure/).

The load-bearing claim: at frontier scale, scaffold variance now exceeds model variance on long-horizon agentic tasks. Reliability lives in the harness, not the model and not the prompt. That makes harness engineering durable platform infrastructure, while prompt engineering and model selection depreciate with each model swap.

## Key concepts

### Scaffold variance beats model variance

The harness-versus-prompt question has a different answer at frontier scale than at the weak-model end, because the failure mode changed. With early models, a harness fixed capability gaps: the model could not reliably follow multi-step instructions. Frontier models almost never fail an individual step. The failure is accumulation over long horizons. Per-step reliability of 99% across 100 steps yields only 36.6% per-task reliability, no matter how good the model is. METR's task-horizon evaluation makes it concrete: frontier models hit roughly 100% on tasks humans finish in under four minutes, and under 10% on tasks that take more than about four hours. Harnesses interrupt the accumulation; prompts cannot.

The benchmark evidence points the same way. Leaderboard analyses of SWE-bench Pro attribute swings of 10 to 20 points to scaffold choices alone, while six frontier models cluster within about a point of each other on the same scaffold. A weaker, cheaper model on a stronger scaffold (Sonnet running a custom harness) has edged the flagship on its own scaffold (Opus) on SWE-bench Pro. Anthropic's engineers name the specific frontier failure modes, premature completion and self-evaluation bias, and describe separating the generator and evaluator agents as the strong lever, not prompt changes.

Two cautions keep this honest. The advantage migrates rather than disappears: each model generation pushes the threshold where a naked loop suffices, so the harness payoff moves to harder, longer tasks. And more harness is not always better. A single 2026 paper reports memory scaffolds degrading performance across all ten frontier models tested, which, if it replicates, is a warning that components should ship with measured reliability deltas rather than intuitions. The cleanest open evidence gap: no public study has measured false-success rate, the agent declaring "done" when it is not, at frontier scale. That is exactly what the verify step addresses.

### Sensors: the underinvested feedback half

Most teams pour energy into guides (the proliferation of markdown rule files) and leave the sensor half, the feedback that catches failures after the agent acts, largely unused, even though computational sensors are deterministic, cheap, and effective against the specific failures agentic coding produces.

The first design decision is informational versus normative. Informational sensors report state ("these dependencies are 18 months old") and leave judgment to the agent or human. Normative sensors render a verdict and can block a build ("dependencies exceed staleness policy, fail"). Most teams stumble on security scanners because a vulnerability check needs semantic judgment, and they wire an informational finding straight to a normative gate with no judgment layer. An LLM sensor can fill that gap, assessing exploitability in context before escalating to a hard block. The split is a per-sensor design choice, not a fixed property.

When an inferential (LLM-based) sensor surfaces the same fuzzy failure repeatedly, encode it as a deterministic rule. The LLM explores the problem space; the deterministic rule enforces the result. That maturity ladder runs inferential-informational, then inferential-normative, then computational-normative, ending at a CI rule that blocks without an LLM in the path. Sensors then live at four points along the pipeline, each with its own cost and coverage: in-session inside the agent loop (cheapest cycle), in CI blocking PRs (the same tools, now gating), continuous drift detection on a cadence (catching debt that accumulates across sessions that each ended clean), and runtime in production (logging and observability checks that become crises when something breaks without useful traces). Designing a harness without placing each sensor on that map is a common omission.

The failures sensors catch are concrete and empirically observed in agent output: too many function arguments, high cyclomatic complexity, long files and methods (agents feel none of the cognitive load a human does), invented module structure, TDD theater (announcing a test-first plan then writing the implementation), assertion-free tests that still hit full coverage, untyped escapes like `no-explicit-any`, and inconsistent imports. Three refinements make sensors better than blunt gates:

- **Custom lint messages as remediation coaching.** A message that says what is wrong and how to fix it in this codebase turns the sensor into a teacher. "Max-lines exceeded. If this file just needs to be slightly longer, raise the threshold for this file from 500 to at most 550 and say why." A compiler error that points at a way forward helps humans and agents alike.
- **Agent-modifiable thresholds.** Per-file relaxations with an inline justification let the agent document its reasoning in the code rather than suppress a rule globally. If the file grows past the new threshold, the sensor re-fires. This makes the exception auditable and models "explain your overrides."
- **A sensor sidecar for supervised sessions.** A parallel process running all sensors gives a human situational awareness (a green/red dashboard) and the agent a single failures-only command to read before committing. Supervised tooling, staying meaningfully in the loop without reading every diff, is underexplored next to the discourse on fully autonomous operation.

Coverage is necessary but not sufficient for test quality. A file can show 100% statement coverage with zero assertions, all of it inherited from a distant end-to-end test. The quality spectrum, by rising rigor and cost:

| Sensor | What it catches | Cost |
|---|---|---|
| Line/branch coverage | Untested paths | $ |
| Property-based testing | Missing logical edge cases | $$ |
| Fuzz testing | Input-resilience gaps | $$ |
| Mutation testing | Weak or missing assertions | $$$$ |

Mutation testing is best reserved for nightly drift detection or pre-merge CI on changed paths, since it is expensive to run in-session.

### Growing a harness: the four-iteration build

The anatomy says what is in a harness. The build pattern (Tejas Kumar, IBM) says how to grow one, in build order, each step adding the smallest component that fixes the current failure. It comes from a live demo that holds the prompt and model fixed and changes only the harness, taking a weak model from "crashes and lies about it" to "reliably succeeds," and it doubles as a retrofit checklist.

1. **Naked loop, add guardrails.** A `while(true)` that calls the model, runs tool calls, and exits on "done" has no resource discipline. Add `maxIterations`, `maxMessages`, naive context compression (keep system, user, last N), and a budget you can defend in a postmortem. Bounded resources; false-success still lurks.
2. **Guardrails, extract a named harness.** A pure refactor: move the loop and guardrails out of the entry point into a `runHarness()`. No behavior change, but now there is a surface to attach more behavior to. Skipping this is why harness logic calcifies and becomes impossible to extend.
3. **Named harness, add a verify step and outer retry.** Wrap the per-attempt loop in a `maxAttempts` retry and require a `verify` function that reflects over the recorded trace, not the agent's claim. Verify is deterministic: it inspects what tool calls fired, what pages the browser actually visited, what the test runner returned. False-success stops; the agent reports failure when it failed.
4. **Verify step, add deterministic interceptors.** For each failure class the verify step surfaces (login walls, missing config, expired tokens), add a per-iteration hook that sees the live world state and either no-ops or executes a deterministic action the agent should not be trusted with.

The ordering is non-obvious. Most teams jump from naked loop straight to interceptors and skip verify, which is the load-bearing piece: without it you cannot tell whether the interceptors helped, and the agent keeps lying about success. The pattern (bound, name, verify, intercept) is the standard maturity ladder for any agent that needs to be reliable rather than impressive, and it is the agent-loop-scoped twin of the workflow-scoped blueprint pattern (deterministic nodes bracketing agentic ones) covered in [Agent Infrastructure](/working-intel/topics/agent-infrastructure/).

### Deterministic interceptors: keep secrets outside the loop

A deterministic interceptor runs inside the agent loop, once per iteration, sees the live world state, and either no-ops or executes a high-trust deterministic action the agent itself should not be trusted with, then injects a synthetic trace message so later iterations see what the harness did. The canonical example is a browser agent hitting a login wall. The naive fixes (credentials in the system prompt, a `login` tool, wrapping the agent in OAuth) all expand the trust boundary to include the LLM. The interceptor keeps it tight: every iteration, the harness reads the current URL; on a login page it fills credentials from an environment variable, submits via raw automation, and tells the agent "handled, continue." The agent never sees the credentials and has no login tool.

This is the cleanest way to map a trust boundary. Secrets (API keys, signing keys, payment tokens) live in interceptor code, not tool definitions or prompts, so prompt injection has nothing to steal. Interceptor actions are deterministic, version-controlled, and reviewable, unlike "the agent decided to log in." Reach for an interceptor when the action involves secrets or high-trust side effects, is deterministic given world state (no agent judgment, just "in situation X, do Y"), and carries asymmetric cost if the agent gets it wrong. Reach for a tool instead when the action needs agent judgment about when or how to act. The split between interceptors and tools is roughly the split between what the harness handles automatically and what the agent decides. Anything touching production data, secrets, payment rails, or compliance-sensitive systems belongs on the interceptor side.

### Verifying non-coding workflows

Coding agents inherit verify primitives for free: lint, type-check, test runner, build. Non-coding workflows (case triage, fraud review, claims processing, contract analysis, support routing) have to design the verify step from scratch, and the published landscape is mature in exactly one narrow domain (RAG QA, via Ragas, DeepEval, TruLens) and fragmented everywhere else. There is no settled catalog of verification patterns. Seven archetypes recur, synthesized from scattered sources rather than a single reference:

1. **Rule-based trace auditor.** Deterministic policy check against the recorded tool-call trace. Best for compliance workflows with enumerable valid actions. Fails on coverage gaps; maintenance grows with workflow complexity.
2. **LLM-as-judge.** A second model scores output against a rubric. Best for open-ended quality. Documented bias modes (positional, verbosity, self-enhancement) make it unreliable when the judge cannot independently verify ground truth.
3. **External oracle.** Output checked against an authoritative source (a database, a system of record, a retrieved document). The strongest primitive where it applies. Limited by oracle freshness.
4. **Self-consistency voter.** Run N times, check agreement. Best for high-stakes single-shot decisions. The trap: if runs share retrieval or systematic bias, consensus signals correlated failure, not correctness. Costs N times.
5. **Rubric-guided decomposition.** Break the task into named sub-goals and verify each. Best for multi-step work with ambiguous end states. Rubric design is load-bearing; an omitted sub-goal lets the verifier sign off on a broken output.
6. **Behavioral compliance monitor.** Verify what the agent did procedurally (tool order, escalation, prohibited actions), not what it said. Best for regulated workflows where process adherence is legally required.
7. **Staged human-in-the-loop sampler.** Route a fraction of outputs to human review by random or confidence-based sampling. Best where full automation is not achievable. Fixed sample rates never catch a systematic error that hits 100% of a case type.

Production designs layer two or three of these (a compliance monitor for process, an external oracle for facts, a sampler for novel patterns), and the layering should be designed at workflow-definition time, not bolted on. Two rules carry weight. Do not default to LLM-as-judge for high-stakes verification, because the judge shares the agent's blind spots; external-oracle verification against a system of record is the better first move where it applies. And note that the verify step is the runtime cousin of the comprehension gate in [Dark Code and the Comprehension Gate](/working-intel/topics/dark-code-and-comprehension-gate/): both reflect over a recorded trace to decide whether output should be trusted, one at PR-review time and one every iteration of an agent loop. The unfilled gap worth naming: contract analysis and case triage need verification of completeness of consideration rather than factual accuracy, and have no published primitive beyond generic judging.

### Self-generated harnesses: state of the field

The forecast where an agent receives a task, generates a task-specific harness, then executes inside what it built is not shipping in production as of mid-2026, but the research is closer than pure speculation. The distinction the literature blurs is workflow planning (mainstream: plan mode) versus harness synthesis (the agent generates its own guardrails, verify functions, and interceptors per task). Only the second is the interesting question. Research clusters into four shapes: meta-harness optimization between runs (AutoAgent, Sakana's Darwin Gödel Machine reporting SWE-bench gains from 20% to 50%, SICA from 17% to 53%); runtime tool synthesis during a task (Live-SWE-agent, arXiv 2511.13646, the closest published match to the forecast); guardrail synthesis from instructions (ContextCov, arXiv 2603.00822, compiling AGENTS.md and CLAUDE.md into executable checks); and joint harness-plus-weight updates (the most ambitious and least replicated). Every published system either pre-optimizes the harness before any task runs or modifies it mid-task; the deliberate "receive task, generate harness, execute" phase is the slot nobody has filled.

The likely production shape, when it lands, is an incremental overlay rather than full generation: a base harness (tool registry, budget limits, logging, safety rails) maintained by a platform team, plus a fast per-task planning call that emits a task-specific diff (which tools to enable, which verify function to apply, which constraint class applies). Designing harness infrastructure as a base layer plus a per-task overlay surface, rather than a monolith per workflow, aligns with that direction and is good engineering regardless. Anthropic's Agent Skills are static today but are the closest production-shipping kin of the overlay model, worth tracking as the spec evolves. The skill-as-overlay framing connects to [Skill Design and Management](/working-intel/topics/skill-design-and-management/), and the meta-agent that edits the harness is the engine behind [Auto-Improving Agents](/working-intel/topics/auto-improving-agents/).

### Operating-model altitude and governance as infrastructure

The same thesis recurs at the organizational scope, which is the altitude a CIO actually decides at. Three operating models for who runs agentic systems: Model A (engineering builds, then hands to IT ops running deterministic runbooks) is structurally incompatible with non-deterministic agents and is where most pilots die. Model B (an embedded "you build it, you run it" pod) removes the handoff gap but breaks past roughly ten pods, when each reinvents a slightly worse auth, observability, and guardrail layer. Model C adds a shared platform (runtime, memory, identity, observability) under the rule that the platform enables and does not constrain: pods keep full autonomy and accountability, but build on a common road. That is the org-scale restatement of the base-layer-plus-overlay thesis, with the platform as the durable base that survives model swaps.

Governance follows the same logic as the deterministic interceptor: policy is enforced outside the LLM loop. You do not ask the agent not to do something; you stop it at the gateway before the model sees the request, which separates who writes policy (security) from who writes the agent (engineering). Policy-as-code is the riverbank, and the agent can meander but cannot cross it. Singapore's IMDA Model AI Governance Framework for Agentic AI (announced January 2026, the first state-backed agentic-specific framework) and Amazon Bedrock AgentCore are framed as converging on the same four pre-action questions: who is the agent and who authorized it, what is it allowed to do, is it performing as expected, and can we audit what it did. Those map onto harness primitives already here: agent identity as a prerequisite, behavioral-compliance monitors for "performing as expected" and "can we audit," and deterministic interceptors for "what it is allowed to do." The matching mental shift is to tolerate variance in execution and stay strict about variance in outcome, which is the operating-model case for the verify step: a fully deterministic agent is just a runbook, so the value is in handling cases you did not anticipate, which is exactly why you verify the outcome rather than constrain the path.

## Current thinking

Harness engineering reads as its own discipline rather than a footnote in agent design, for three reasons. The empirical leverage is real: scaffold choices dominate model choices in expected outcome on the benchmarks that exist. The investment is durable, where prompt tuning depreciates with each model swap, which argues for budgeting harness work as platform engineering. And the trust-boundary mapping (the deterministic-interceptor pattern) is the cleanest way to keep secrets and high-trust actions outside the model's context, which matters more in regulated settings than generic agent-design advice captures. The throughline across every scope, agent loop, workflow, and organization, is the same: put the durable, deterministic structure in the envelope, let the model be non-deterministic inside it, and verify the outcome rather than trusting the claim. The reliability story this serves at the production layer (behavioral failure, tracing, evaluation as the spec) lives in [Taking Agents to Production](/working-intel/topics/taking-agents-to-production/), and the context-construction component of the envelope is the discipline in [Context Engineering](/working-intel/topics/context-engineering/).

## Open questions

- Where exactly on the task-length axis does a naked loop stop sufficing for a given model generation? Each model swap moves the threshold, and the labs hold the calibration data internally. A public map for current frontier models would be valuable.
- Does the guides-versus-sensors substitution hypothesis hold? Once a sensor covers a failure mode, does the corresponding guide become redundant? Testable by adding the sensor and checking whether the guide still fires.
- What is the optimal sensor density? Over-active sensors cause thrashing (an aggressive max-lines rule pushed an agent to over-fragment files); too few let quality drift. The signal-to-noise tradeoff is uncharacterized.
- Does the memory-scaffold-degradation finding replicate? If true, it is a real design constraint, since not all harness components help. Single source so far.
- Can false-success rate be instrumented in real agent workflows? The literature has no quantitative measure of "agent declares done when it is not" at frontier scale.
- Can a behavioral-compliance monitor be derived automatically from a written governance standard, the way ContextCov compiles instruction files into executable guardrails?
- Where do skills sit in the harness anatomy? They look like a hybrid of tool registry, prompt fragment, and workflow contract, and the synthesis research hints they become the per-task overlay surface.
