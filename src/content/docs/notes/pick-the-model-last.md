Sandipan Bhaumik, a data and AI lead at Databricks who spent five years as a principal architect at AWS, has watched the same project die over and over. Leadership feels pressure to ship something with AI. The first meeting opens with "shall we use GPT or Claude?" A team picks a model, builds features, demos them on clean data in a controlled environment, and leadership signs off. A few weeks into production the questions start: "what the hell is AI doing?" The demo never scaled, and the money spent building it returned nothing.

From a couple of years of these failures across B2B and regulated-finance clients, Bhaumik pulled out three gaps that sink production AI: an observability gap (you cannot see what the agent did), an evaluation gap (you never defined the one number that means success for the business), and a governance gap (nobody owns the failure at 3 a.m.). His fix is a five-pillar playbook (evaluation, observability, data foundation, orchestration, governance) and one reordering that carries the whole argument. In his eight-week banking-chatbot build, the team chose the model in week seven.

That inversion sounds reckless and turns out to be the disciplined move. Weeks one and two went to evaluation, which Bhaumik treats as the specification for the system. The team sat with human support agents, collected 200 real answers to the simple queries a chatbot should deflect, wrote down what happens in the confusing edge cases, and turned that into a golden dataset with a hard target: deflect 60% of queries at roughly 85% accuracy. Then they automated the comparison so live responses score against the set continuously. Only after observability and a data foundation were in place did model choice come up, and by then it was a quick measurement against the dataset rather than a month of debate. An earlier vendor POC had burned about 85,000 dollars over six months with none of this, and no one could say why it failed.

## Failures are behavioral, so reliability is not a model property

The reason the order works is that production agents rarely fail at the model. They fail across the trajectory. Bhaumik's example: a user asks for an account balance, the agent returns the right number, and the answer passes every check. Walk the trace and you find the agent made three duplicate database calls to get there, retrying through failures. Harmless in a demo, expensive across thousands of daily queries.

The wider field has landed in the same place. A 2026 production guide from Confident AI puts it bluntly: "An agent can look busy, reason intelligently, call the right-looking tools, and still fail to complete the task," and so "LLM agent evaluation is how you prevent those failures before they compound in production." Google Cloud names the exact pattern a "silent failure," where "an agent tasked with reporting inventory might give the correct number but reference last year's report by mistake. The result looks right, but the execution failed." A practitioner write-up on the 2026 deployment loop catalogs where agents break: bad retrieval on step 2, wrong tool arguments on step 4, silent state corruption on step 5, a plausible-looking final answer on step 8. None of those is a model-quality problem. All of them are invisible without a trace.

This is why Bhaumik treats his third evaluation layer, the behavioral one, as the layer teams skip and the one that matters most. The first layer is deterministic (regex, format and PII checks, classic ML for intent). The second is semantic (an LLM judge scoring groundedness and relevance against the golden answers). The third watches the agent's process itself: which tools it called, whether it looped, how many API hits one answer cost. Confident AI splits the same layer into tool correctness, argument correctness, and step efficiency. Google Cloud frames it as testing the trajectory, not the final output. The convergence is the point. Once you accept that failures are behavioral, reliability stops being something you buy with a better model and becomes something you build with a measurement system.

## The measurement system is a living, owned asset

The order also explains why observability is not optional. In a regulated bank, when a customer disputes a fee waiver, you need to show what the agent did at each step, and Bhaumik notes that European and other regulators now refuse to onboard agents to production without it. Tracing pays off twice more: it turns into online monitoring that can cap retries or fall back to a human mid-incident, and it turns production failures into new test cases. A few weeks after launch, the chatbot's satisfaction scores dropped. The trace showed the agent reading a policy document the bank had already superseded, because the new embeddings never made it into the vector database. Without the measurement system, that reads as "AI is acting up." With it, it reads as a stale-index bug with a fix.

That feedback loop is the same one every serious source describes: convert each failure into a test case so the bug stays fixed, and gate deployments on the growing suite. Bhaumik calls the dataset a living system that starts around 200 cases and only gets better as it grows. Google Cloud adds a useful caveat to the golden-dataset orthodoxy: you do not always need a pre-labeled set to start, since you can synthesize cases with dueling LLMs or curate anonymized production sessions. The deployment-loop writers compress it to one line: "observability without evals produces dashboards / evals without observability produce blind benchmarks."

A living asset needs an owner, which is where Bhaumik's last lessons land. The test library needs categories and an accountable owner or it rots. Prompt changes need real change-management discipline, not one-word git commits, so you can trace why a prompt changed and what failure it fixed. Model upgrades need to run against your dataset before you trust them, which keeps you free to switch providers. And an incident playbook (detect on the eval dashboard, diagnose in the trace, contain by reverting the prompt or routing to a human, fix and add the case back, then alert through your existing ITSM) turns "AI broke" into an operational runbook.

The lesson under the playbook is the one worth keeping. The model question feels urgent because the model is the new and exciting part. The work that decides whether an agent survives production is the measurement and governance system around it, and that work is mostly the unglamorous engineering teams already know how to do. Build it first, and the model becomes a swappable, last decision.
